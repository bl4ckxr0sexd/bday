# 🎉 Birthday Invitation Workflow

A reusable, three-part invitation flow. Edit **one file** (`.env`), run the build,
and everything updates.

```
Email (email.html) ──▶ Landing page (index.html) ──▶ Google Drive download
                              │
                              └──▶ tracking event ──▶ Apps Script (webapp.js) ──▶ Google Sheet
```

## Project structure

```
birthday/
├── .env                  ← THE ONE FILE YOU EDIT (names, links, party details)
├── .env.example          ← documented reference copy
├── build.js              ← injects .env into the templates (no dependencies)
├── package.json          ← `npm run build`
├── vercel.json           ← Vercel build config
├── webapp.js             ← Google Apps Script (the tracking receiver)
├── templates/
│   ├── index.template.html   ← landing page source
│   └── email.template.html   ← email source
└── public/               ← GENERATED by the build (index.html, email.html)
```

You never edit files in `public/` — they're regenerated from `templates/` + `.env`.

## Configurable values (`.env`)

| Key                 | What it controls                                            |
|---------------------|------------------------------------------------------------|
| `COUSIN_NAME`       | Birthday person's name (email + landing page)              |
| `GUEST_NAME`        | Email greeting, e.g. "there" → "Hi there! 👋" (or a merge tag) |
| `LANDING_URL`       | Landing page URL (used in the email button)                |
| `DOWNLOAD_URL`      | Google Drive link for the downloadable invitation          |
| `TRACKING_ENDPOINT` | Apps Script Web App URL (`/exec`). Blank = tracking off     |
| `PARTY_DATE` / `PARTY_DATE_NOTE` | e.g. "July 15, 2026" / "Wednesday"            |
| `PARTY_TIME` / `PARTY_TIME_NOTE` | e.g. "4:00 PM" / "Onwards"                    |
| `DRESS_CODE` / `DRESS_CODE_NOTE` | e.g. "Bright & Colorful" / "..."              |

> `GUEST_NAME` is a fixed greeting by default ("there"). For per-recipient
> personalisation, set it to your email tool's merge tag (e.g. `*|FNAME|*`).

## Setup — do these in order

### 1. Deploy the data receiver (`webapp.js`)
Follow the comment block at the top of `webapp.js`. You'll get a **Web App URL**
ending in `/exec`. Put it in `.env` as `TRACKING_ENDPOINT`.

### 2. Fill in `.env`
Set `COUSIN_NAME`, `DOWNLOAD_URL`, party details, etc. (`LANDING_URL` comes from step 3.)

### 3. Deploy the landing page to Vercel
`vercel.json` already tells Vercel to run the build and serve `public/`.

**CLI**
```bash
npm run build        # optional: preview locally by opening public/index.html
vercel               # deploy a preview
vercel --prod        # promote to production
```

**Dashboard:** push this folder to a Git repo and "Import Project" on
[vercel.com](https://vercel.com), or drag-and-drop the folder. Vercel runs
`node build.js` and serves `public/` automatically.

Copy your live URL, put it in `.env` as `LANDING_URL`, and redeploy so the
email's button points to it.

### 4. Send the email
The built email is at `public/email.html` and greets the reader using
`GUEST_NAME` from `.env` (default "there"). Paste the HTML into your email
tool's "code/HTML" mode and send.

## Local development
```bash
npm run build         # regenerate public/ from templates + .env
open public/index.html
```
Change `.env` or a template, run `npm run build` again, refresh.

## Notes
- **Zero dependencies** — the build is plain Node, so Vercel builds fast.
- **Tracking** collects coarse network info (IP / ISP / country) for guest
  analytics only. If `ipapi.co` is rate-limited the page still loads and still
  sends events (with `unknown` network fields).
- `public/` and `.env` handling are covered in `.gitignore` — `.env` is safe to
  commit for this template (no real secrets).
